**Parent Scaffold context (verified at planning time):**
The TransactionsScreen is hosted by `MoneyManagerNavHost.kt` at line 232:
```kotlin
Scaffold(
    bottomBar = {
        NavigationBar { ... NavigationBarItems ... }
    }
) { innerPadding ->
    NavHost(..., modifier = Modifier.padding(innerPadding))
}
```

Material 3's `NavigationBar` composable applies `windowInsets = NavigationBarDefaults.windowInsets` (which equals `WindowInsets.navigationBars`) by default and consumes that inset upward. Therefore the `innerPadding` passed to NavHost (and onward to TransactionsScreen via NavHost's destination padding) **already accounts for both the NavigationBar height and the system nav bar inset** as a combined bottom padding.

This means: applying `Modifier.padding(padding)` on the LazyColumn (already done) handles the bottom inset. Adding `WindowInsets.navigationBars.asPaddingValues()` as `contentPadding` would **double-count** the nav bar inset, contradicting D-05's "no gap" requirement.
